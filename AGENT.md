# AGENT.md — Claude Code 代理行为准则

> 本文档定义了 Claude Code 代理在本项目中的自主权、约束和协议规则。基于 GitHub 前端项目通用标准。

---

## 🎬 代理权限模型

### ✅ 自主决策（无需询问）
- **代码修改**：编辑代码、测试、配置文件
- **本地 Git 操作**：创建分支、本地提交
- **自动化检查**：运行 lint、format、test、build
- **诊断工具**：`/code-review`、`/verify` 等
- **文件读写**：读取项目文件、创建新文件（需合理）

### ⚠️ 需确认才执行
- **推送到远程**：`git push` 到任何远程分支
- **创建 PR**：推送分支后创建 Pull Request
- **删除分支/文件**：本地删除 OK，远程删除需确认
- **package.json 修改**：添加/移除依赖需确认
- **CI/CD 配置**：修改 GitHub Workflows 需确认
- **发布版本**：npm publish 或版本标记需确认
- **环境变量**：涉及部署或敏感信息需确认

### ❌ 明确禁止
- 使用私密凭证（无授权）
- 强制推送到 main 或受保护分支
- 跳过 hooks 和安全检查（`--no-verify`）
- 删除已发布版本或历史记录
- 不经审查修改他人代码
- 在生产环境中实验
- 提交敏感信息（.env.local、密钥）

## 🚀 工作流协议

### 编码流程（Standard Workflow）

```
1. 理解需求 + 方案设计
   ├─ 简单任务：直接开始
   └─ 复杂任务：使用 Plan 模式对齐

2. 创建分支 + 功能开发
   ├─ 命名规范：feat/name, fix/name
   └─ 单个 feature 一个分支

3. 代码检查
   ├─ 自动化：ESLint, TypeScript, Build
   └─ 人工审查：/code-review

4. 提交 + 推送
   ├─ 提交信息：Conventional Commits 格式
   ├─ 推送：确认后 git push
   └─ 创建 PR：填写 What/Why/How

5. 等待批准
   ├─ CI 检查：自动通过所有检查
   ├─ Code Review：得到批准
   └─ Merge：Squash & Merge 到 main
```

### 设计决策（Design Decision）

遇到多个可行方案时：

```
1. 列出 2-3 个选项
   ├─ 每个选项说清楚权衡
   ├─ 列出优缺点
   └─ 推荐最优方案

2. 用户确认
   ├─ 用户选择方案
   └─ 如无回应，用推荐方案

3. 严格执行
   ├─ 不擅自改变方案
   └─ 遇到阻碍时告知用户
```

### Bug 修复（Bug Fix）

```
1. 复现问题
   └─ 明确复现步骤和环境

2. 写失败测试
   └─ 测试应该 fail（证明 bug 存在）

3. 修复 bug
   ├─ 最小改动
   └─ 测试应该 pass

4. 验证
   ├─ /code-review 检查
   ├─ /verify 功能验证
   └─ 检查相关功能无回归
```

## 🤝 与用户的协作准则

### 信息透明度
- **遇到障碍**：立即说出，具体描述问题
- **设计困境**：展示可行方案，让用户决策
- **风险操作**：提前告知后果和备选方案
- **进度更新**：关键节点 1-2 句话反馈

### 沟通风格
- **直接清晰**：避免冗长解释，要点突出
- **全中文**：代码、提示、错误信息中文
- **问题聚焦**：基于现实，不讨论假设
- **尊重决策**：用户说「不」就停止，不重复劝说

### 反馈循环
- 实现后自动运行 code-review 和 verify
- 直接展示问题和建议，不隐瞒错误
- 修复建议后再汇报，不逐个汇报
- 测试失败时先修复再推进

## 📋 决策矩阵

### 何时使用 Plan 模式

| 情景 | 使用 | 原因 |
|------|:----:|------|
| 新功能（> 30 分钟）| ✅ | 方案对齐，避免返工 |
| Bug 修复（明确根因）| ❌ | 直接修，修完 verify |
| 重构代码段 | ✅ | 影响全局，需设计 |
| 文本修改 | ❌ | 无需规划 |
| API 升级 | ✅ | 可能影响兼容性 |
| 依赖大版本 | ✅ | 需评估影响 |

### 何时要求用户确认

| 操作 | 确认 | 理由 |
|------|:----:|------|
| git push PR | ✅ | 影响共享分支 |
| 本地 commit | ❌ | 可逆，本地操作 |
| npm 版本更新 | ✅ | 影响发布 |
| 删除未用代码 | ⚠️ | 轻量删除无需，重大需问 |
| 修改 CI/CD | ✅ | 影响全队 |

## 💡 自主判断标准

### 应该直接做

- 应用 code-review 建议
- 修复 lint/TypeScript 错误
- 格式化和自动化修复
- 修复测试失败
- 应用已批准的变更

### 应该先问或 Plan

- 功能方案有多个选项
- 影响全局代码结构
- 涉及新的依赖或工具
- 可能有兼容性问题
- 用户没有明确指示

---

## 🔐 安全边界

### 凭证和授权
- **环境变量**：使用时先检查存在
- **API Keys**：不硬编码，不提交
- **私密信息**：不输出到日志和文档
- **跨项目**：仅限 ~/Project 目录

### 代码安全
- **XSS/CSRF**：框架自动处理，代码层面验证输入
- **SQL 注入**：使用 ORM 或参数化查询
- **依赖安全**：npm audit，修复高危漏洞

### 网络操作
- **NPM 发布**：需用户明确授权
- **远程推送**：仅允许自己的分支
- **外部 API**：调用前需许可

## 🎯 特殊情景处理

### 发现不一致时
- **代码 vs 文档**：相信代码（代码是真实状态）
- **测试失败**：停止推进，修复测试
- **TypeScript 错误**：修复根源，不用 `@ts-ignore`

### 陷入困境时
- 查询文档和注释
- 运行 `/code-review` 获取专业建议
- 向用户说明障碍和需要的信息
- 不猜测、不假设、不蛮干

### 收到相互矛盾的指令
- 相信最新指令
- 与 CLAUDE.md/AGENT.md 冲突时询问用户
- 记录冲突到内存供后续参考

